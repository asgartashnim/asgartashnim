## Detection Context

This post documents a Sigma rule contributed to [SigmaHQ](https://github.com/SigmaHQ/sigma) targeting
post-phishing inbox rule abuse in Microsoft 365 Exchange Online.

**Log Source:** M365 Unified Audit Log  
**Technique:** MITRE ATT&CK T1114.003 — Email Collection: Email Forwarding Rule  
**Status:** Experimental — pending production validation  

---

## The Technique

After a successful phishing attack, credential theft alone is rarely the end goal. Persistent
access to a victim mailbox is more valuable — and inbox rules are how attackers get it.

The pattern is consistent across post-phishing IR cases:

1. Attacker obtains credentials via phishing
2. Attacker authenticates to Exchange Online
3. Attacker creates an inbox rule that forwards all email to an external address
4. Attacker optionally configures the rule to delete originals or mark them as read
5. Attacker now has persistent visibility into the victim's inbox — one that survives a password reset

The rule persists because most organizations don't audit inbox rule creation, and victims rarely
notice because the email still arrives normally. The forwarding happens silently in the background.

This maps to two MITRE techniques simultaneously — T1114.003 (collection via forwarding rule) and
T1070 (defense evasion via message deletion).

---

## The Detection Gap

SigmaHQ had no coverage for this technique in the M365 Unified Audit Log. One existing rule
(`microsoft365_susp_inbox_forwarding.yml`) detects Microsoft Cloud App Security alerts — meaning
it only fires after Microsoft's own anomaly detection has already flagged something. That rule
requires a Defender for Cloud Apps license and depends entirely on Microsoft's alerting pipeline.

This rule operates at the audit log layer directly — no dependency on Microsoft's detection engine,
no license requirement beyond basic M365 auditing. It fires on the raw cmdlet execution event.

---

## Detection Logic

The M365 Unified Audit Log records Exchange Online PowerShell operations. When an inbox rule is
created or modified, it generates an event with two relevant fields:

- `Operation` — the cmdlet executed (`New-InboxRule` or `Set-InboxRule`)
- `Parameters` — the arguments passed to the cmdlet as a JSON array

The rule targets `Operation` for rule creation or modification, then checks `Parameters` for
forwarding targets (`ForwardTo`, `ForwardAsAttachmentTo`, `RedirectTo`) or suppression behavior
(`DeleteMessage`, `MarkAsRead`).

```yaml
title: Suspicious Exchange Inbox Rule Configured for External Forwarding or Message Suppression
id: 756232a0-5b3d-4665-86a1-12187c1bd427
status: experimental
description: |
    Detects creation or modification of Exchange Online inbox rules that configure
    external email forwarding, redirection, or message deletion. This technique is
    consistently observed as a post-compromise persistence and collection mechanism
    following successful phishing attacks.

    After gaining access to a victim mailbox, adversaries routinely create inbox rules
    to silently forward email to attacker-controlled external addresses (T1114.003) and
    optionally suppress evidence by deleting the originals from the victim inbox. These
    rules persist across password resets unless explicitly removed, making them a high-value
    indicator of account compromise.
references:
    - https://learn.microsoft.com/en-us/security/operations/incident-response-playbook-phishing
    - https://attack.mitre.org/techniques/T1114/003/
    - https://attack.mitre.org/techniques/T1070/
author: Tashnim Asgar (lanceterminal)
date: 2026/04/08
tags:
    - attack.collection
    - attack.t1114.003
    - attack.defense_evasion
    - attack.t1070.008
logsource:
    product: m365
    service: exchange
detection:
    selection_operation:
        Operation:
            - 'New-InboxRule'
            - 'Set-InboxRule'
    selection_forwarding:
        Parameters|contains:
            - 'ForwardTo'
            - 'ForwardAsAttachmentTo'
            - 'RedirectTo'
    selection_suppression:
        Parameters|contains:
            - 'DeleteMessage'
            - 'MarkAsRead'
            - 'MoveToFolder'
    condition: selection_operation and (selection_forwarding or selection_suppression)
falsepositives:
    - Legitimate users configuring email forwarding to a personal or secondary account
    - IT or compliance-driven forwarding rules for archival or legal hold purposes
    - Helpdesk-created rules during mailbox migration or consolidation workflows
level: medium
```

## Converted Query — KQL (Microsoft Sentinel)

```kql
(Operation in~ ("New-InboxRule", "Set-InboxRule")) and 
((Parameters contains "ForwardTo" or Parameters contains "ForwardAsAttachmentTo" or Parameters contains "RedirectTo") 
or 
(Parameters contains "DeleteMessage" or Parameters contains "MarkAsRead"))
```

The `in~` operator performs a case-insensitive match. The `contains` operator handles substring
matching against the Parameters field, which Microsoft serializes as a JSON array in the audit log.

## Investigation Pivots When This Fires

When this rule generates an alert, the first priority is determining whether the forwarding target
is internal or external, and whether the rule was created with or without a corresponding change
management record.

Start here:

```powershell
# Check for delegated access changes on the mailbox
Get-MailboxPermission -Identity <user>

# Check if the rule was deleted and recreated to evade detection
Search-UnifiedAuditLog -Operations Remove-InboxRule -UserIds <user>

# Check if SMTP forwarding is also configured at the mailbox level
Get-Mailbox <user> | Select ForwardingSmtpAddress, DeliverToMailboxAndForward
```

Cross-reference the `ClientIP` from the audit log event against Microsoft Entra sign-in logs for
the same user. Look for anomalous geography, unfamiliar UserAgent strings, or legacy authentication
protocols — these indicate the session that created the rule may itself be the result of credential
theft or token abuse.

Rules combining `ForwardTo` and `DeleteMessage` with no prior business justification should be
escalated regardless of account context. The suppression behavior is the high-confidence indicator.

---

## Limitations

This rule is experimental and has not been validated against real M365 audit log events in a
production environment. The `Parameters` field structure is assumed to contain the parameter names
as searchable strings — this needs to be confirmed against a real audit log event before deploying
in a production SIEM.

False positive volume will be environment-dependent. Organizations with active mailbox migrations,
shared mailbox configurations, or permissive forwarding policies will need to tune aggressively
before this rule is operationally useful.
