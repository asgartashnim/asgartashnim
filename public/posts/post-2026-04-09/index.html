Zero-Trust Hardening a Proxmox Hypervisor to a Single UDP Port

April 9, 2026 - 9 mins read
  • SSH Hardening
  • Zero Trust Network
  • Bash Script

Overview

A default Proxmox VE installation exposes four services on the public internet. The Proxmox web UI (port 8006), SSH (port 22), rpcbind (port 111), and the SPICE proxy (port 3128) are all reachable from anywhere on the internet with no network-layer protection. This post documents a hardening approach that reduces the public attack surface to a single UDP port using Tailscale as a zero-trust access layer and iptables to enforce the network boundary at the kernel level.

The Baseline Problem

To quantify the threat, authentication logs from a standard datacenter server were captured over a 72-day observation period before any hardening was applied. The server had a public IP with port 22 open and no special exposure or reason to be targeted specifically.

Metric Value
Total failed password attempts 572,212
Unique attacker IPs 6,847
Average attempts per day 7,947
Average attempts per hour 331
Average attempts per minute 5.5
Average attempts per attacker IP 83.5

This is not targeted attack activity. This is the baseline noise that every internet-facing server receives. Automated botnets scan the entire IPv4 address space continuously. A server with port 22 open on a public IP will receive brute force attempts within minutes of provisioning.

The top 10 attacker IPs by attempt volume accounted for 19.4% of all attempts. IPs ranked 6 through 10 all shared the 45.148.10.x subnet, indicating coordinated botnet infrastructure operating from a single network block.

The most targeted usernames — admin (60,711 attempts), ubuntu (33,980), postgres (17,785) — reflect systematic cycling through default and service account names common in Linux environments.

At this baseline, a real attacker with compromised credentials would be completely invisible in the noise. Detection is effectively impossible when the false positive rate is 5.5 events per minute from 6,847 distinct sources.

Pre-Hardening Port Exposure

Running ss -tlnp on a fresh Proxmox install reveals the following listening services:

Port Service Bound To Exposure
22 sshd 0.0.0.0 Public internet
111 rpcbind 0.0.0.0 Public internet
8006 pveproxy (Web UI) * Public internet
3128 spiceproxy * Public internet
25 postfix 127.0.0.1 Localhost only
85 pvedaemon 127.0.0.1 Localhost only

Port 8006 is the Proxmox management interface. It is the most significant exposure — direct access to the hypervisor management plane from anywhere on the internet with nothing between it and the scanner.

Architecture

The hardening approach operates on two layers that protect against different threat scenarios.

Network layer — iptables

iptables enforces a default DROP policy on the INPUT and FORWARD chains. All inbound traffic is blocked at the kernel level before any application processes it. The only exception is UDP port 41641, which is required for Tailscale’s WireGuard handshake. Every other port — including SSH on 22 and the Proxmox web UI on 8006 — becomes unreachable from the public internet.

Access layer — Tailscale

Tailscale provides zero-trust remote access. It is WireGuard under the hood with a managed control plane that handles key exchange, device authentication, and routing. Each device on the Tailnet authenticates cryptographically — there is no password to brute force and no service to probe. The Proxmox server is reachable only from authenticated devices on the Tailscale network, over the encrypted WireGuard tunnel through UDP 41641.

Together, these layers mean the public IP is dark to everyone except devices that have been explicitly authorized in the Tailscale admin console.

Implementation

IP Forwarding

IP forwarding must be enabled at the kernel level before Tailscale can route subnet traffic or function as an exit node:

echo 'net.ipv4.ip_forward = 1' > /etc/sysctl.d/99-tailscale.conf
echo 'net.ipv6.conf.all.forwarding = 1' >> /etc/sysctl.d/99-tailscale.conf
sysctl -p /etc/sysctl.d/99-tailscale.conf

Written to /etc/sysctl.d/ rather than /etc/sysctl.conf for persistence across reboots without modifying the base configuration file.

Disabling rpcbind

rpcbind serves the RPC portmapper function for NFS and other RPC-based services. A standalone Proxmox host has no requirement for it. It was listening on port 111 on all interfaces:

systemctl stop rpcbind rpcbind.socket
systemctl disable rpcbind rpcbind.socket

SSH Hardening

SSH hardening is applied independently of the firewall rules — defense in depth ensures that even if the firewall were bypassed, the authentication layer remains secure:

PermitRootLogin prohibit-password
PasswordAuthentication no
PubkeyAuthentication yes
MaxAuthTries 3
LoginGraceTime 30
ClientAliveInterval 300
ClientAliveCountMax 2
X11Forwarding no
AllowAgentForwarding no
AllowTcpForwarding no

MaxAuthTries 3 reduces the attempts available per connection from the default of 6. LoginGraceTime 30 cleans up half-open connections from scanners faster than the default 120 seconds. All forwarding options are disabled — SSH is used for interactive shell access only.

The configuration is validated with sshd -t before restarting the daemon. If validation fails, the original configuration is restored from a timestamped backup and the process exits. This prevents locking out remote access due to a config error.

iptables Ruleset

# Default policies
iptables -P INPUT   DROP
iptables -P FORWARD DROP
iptables -P OUTPUT  ACCEPT

# Allow established/related connections
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow all traffic from the Tailscale interface
iptables -A INPUT -i tailscale0 -j ACCEPT

# Allow Tailscale to forward packets — required for exit node and subnet routing
iptables -A FORWARD -i tailscale0 -j ACCEPT
iptables -A FORWARD -o tailscale0 -j ACCEPT
iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT

# NAT masquerade — required for exit node internet routing
iptables -t nat -A POSTROUTING -o vmbr0 -j MASQUERADE

# Allow Tailscale WireGuard handshake — only public port needed
iptables -A INPUT -p udp --dport 41641 -j ACCEPT

# Log dropped packets — rate-limited to prevent log flooding
iptables -A INPUT -m limit --limit 5/min --limit-burst 10 \
  -j LOG --log-prefix "iptables-DROP: " --log-level 4

The MASQUERADE rule requires explanation. When the server functions as a Tailscale exit node, packets arrive from devices’ private Tailscale IPs in the 100.x.x.x range. These addresses are not routable on the public internet. Without MASQUERADE, outbound packets leave with a private source IP and the response has nowhere to return. MASQUERADE rewrites the source IP on outgoing packets to the server’s public IP, ensuring return traffic routes correctly back through the tunnel.

The FORWARD ESTABLISHED,RELATED rule permits return traffic through the forward chain. Without it, outbound requests succeed but the responses are dropped by the default FORWARD DROP policy — connections appear to hang silently.

Rules are persisted via iptables-persistent so they survive reboots.

UDP GRO Optimization

Tailscale warns when UDP GRO forwarding is not optimized on Proxmox bridge interfaces. Applying the optimization improves WireGuard throughput, which is particularly relevant when the server is used as an exit node:

ethtool -K vmbr0 rx-udp-gro-forwarding on rx-gro-list off

This is applied with a graceful fallback — if the NIC driver does not support the feature, the script continues normally rather than failing.

Secret Management

Sensitive values — the SSH public key and Tailscale auth key — are passed as environment variables rather than hardcoded in the script. The script file contains no credentials and is safe to publish to a public repository.

# Leading space prevents lines from being saved to bash history
 export SSH_PUBKEY="ssh-ed25519 AAAAC3..."
 export TAILSCALE_AUTHKEY="tskey-auth-..."

bash proxmox-zerotrust-deploy.sh

# Clear session history before exiting
cat /dev/null > ~/.bash_history && history -c

The ${VAR:?error} bash expansion syntax is used for required variables. If either is unset, the script exits immediately before making any system changes:

proxmox-zerotrust-deploy.sh: SSH_PUBKEY: ERROR: SSH_PUBKEY
environment variable is not set. Export it before running.

Variables with safe defaults use ${VAR:-default} syntax, allowing runtime override while providing sensible fallback values.

Verification

The script runs 13 automated checks before declaring the deployment complete:

[OK]    tailscaled service: running
[OK]    Tailscale IP: 100.x.x.x
[OK]    IP forwarding: enabled
[OK]    rpcbind: inactive
[OK]    iptables INPUT policy: DROP
[OK]    iptables FORWARD policy: DROP
[OK]    iptables tailscale0 INPUT rule: present
[OK]    iptables tailscale0 FORWARD rules: present
[OK]    iptables UDP 41641 rule: present
[OK]    Exit node: advertised
[OK]    SSH authorized_keys: present (1 key(s))
[OK]    SSH PasswordAuthentication: disabled
[OK]    iptables-persistent: installed (rules survive reboot)

All verification checks passed.

Any failed check prevents the reboot prompt from appearing. The deployment is not considered complete until all 13 checks pass.

Post-Hardening Results

An external nmap scan after deployment:

nmap -Pn [REDACTED]

Not shown: 65532 filtered tcp ports (no-response)
PORT    STATE  SERVICE
113/tcp closed ident

Port 113 returns closed rather than filtered — this is the kernel sending a TCP RST directly, which is standard Linux behavior for the ident service and does not represent an exploitable surface. All other 65,532 TCP ports are filtered. The Proxmox web UI on 8006 and SSH on 22 are completely unreachable from the public internet.

The iptables drop log immediately began recording blocked attempts:

iptables-DROP: SRC=195.178.110.26 DPT=22
iptables-DROP: SRC=98.209.242.15  DPT=8006

195.178.110.26 was the second-highest volume attacker in the pre-hardening baseline data with 18,379 recorded attempts. The same infrastructure is still scanning. The outcome is now different.

Detection Engineering Implications

This hardening applies a core detection engineering principle at the network layer: reduce the baseline until anomalies become visible.

Before hardening, the signal-to-noise ratio made meaningful detection impossible. 5.5 automated attempts per minute from 6,847 distinct sources generates a background radiation that obscures real attacker activity completely.

After hardening, the auth log baseline is zero. SSH is unreachable from the public internet. Any entry in the auth log represents traffic that arrived through the only permitted path — an authenticated Tailscale device.

Alert Severity Model

Finding Severity Meaning
Auth log entry matching known Tailscale IP and expected time None Normal authenticated access
Auth log entry from unrecognized Tailscale IP Critical Tailscale account compromised or unauthorized device added
Auth log entry at unexpected time from known IP High Device may be compromised and used as pivot
Any SSH activity on the public IP Critical iptables rules failed or bypassed

The PrintLastLog yes sshd directive provides a passive tripwire. Every login displays the previous login timestamp and source IP. A discrepancy between the displayed values and the operator’s last known session is an immediate compromise indicator visible before any diagnostic commands are run.

Monitoring Commands

# Review login history
last

# Monitor auth log in real time
tail -f /var/log/auth.log | grep sshd

# Audit Tailscale device inventory
tailscale status

# Verify firewall policy is still in place
iptables -L INPUT -n | head -3
# Expected: Chain INPUT (policy DROP)

# Verify public IP is dark from outside Tailscale
curl -k --connect-timeout 10 https://<PUBLIC-IP>:8006
# Expected: curl: (28) Connection timed out

Conclusion

Reducing the public attack surface of a Proxmox hypervisor from four exposed services to a single UDP port eliminates the vast majority of automated opportunistic attack volume. The approach is not complexity — it is constraint. Every unnecessary service removed, every unnecessary port closed, every unnecessary credential path eliminated narrows the window an attacker needs to find something exploitable.

The same principle that makes this hardening effective — reducing the baseline until anomalies are visible — is the foundation of effective detection engineering at every layer of the stack.
